Race Conditions
em Transações Bancárias

Relatório de Experimento Computacional

Avaliação comparativa de seis mecanismos de sincronização — primitivos POSIX Pthreads na camada de aplicação e controle transacional no PostgreSQL — aplicados a transferências bancárias concorrentes. O experimento quantifica o impacto de condições de corrida e mede o desempenho de cada solução.

1. Introdução

Imagine dois caixas de banco que, ao mesmo tempo, consultam o saldo da mesma conta: os dois veem R$ 1.000, cada um calcula de forma independente e os dois gravam o resultado. O segundo caixa grava por cima do primeiro — como se a operação anterior nunca tivesse existido. Esse é o problema central deste trabalho: quando múltiplas threads (linhas de execução que rodam em paralelo) acessam o mesmo dado sem coordenação, o resultado depende da ordem aleatória em que o sistema operacional as escalona. Esse fenômeno é chamado de condição de corrida (race condition).

Em sistemas financeiros, o efeito é silencioso e perigoso: dinheiro pode ser criado do nada ou desaparecer sem nenhuma mensagem de erro. O programa executa normalmente, os registros mostram tudo correto — mas os saldos estão errados. Este experimento reproduz esse cenário em escala e compara seis mecanismos de proteção diferentes.

02 pthread_mutex_t {{SCN_2_BADGES}}
{{SCN_2_DYNAMIC}}

Mecanismo

Funciona como a chave de um banheiro compartilhado: quem pegar a chave (pthread_mutex_lock) executa a operação; os outros esperam do lado de fora. Quando termina, devolve a chave (pthread_mutex_unlock) e a próxima thread pode entrar. Como há uma única chave para todas as contas — independente de quais contas a transferência envolve — nenhuma operação ocorre em paralelo. Isso garante corretude, mas elimina o paralelismo. 

pthread_mutex_lock(&g_lock);    // pega a chave — bloqueia se outra thread a tem
    bf = pg_read_bal(c, from);
    bt = pg_read_bal(c, to);
    pg_write_bal(c, from, bf - amt);
    pg_write_bal(c, to,   bt + amt);
pthread_mutex_unlock(&g_lock);  // devolve a chave — libera para a próxima thread

Avaliação

03 sem_t — Semáforo POSIX {{SCN_3_BADGES}}
{{SCN_3_DYNAMIC}}

Mecanismo

Pense num estacionamento com painel de vagas: quem entra decrementa o contador (sem_wait); quem sai o incrementa (sem_post). Se o painel marca zero, o carro aguarda na entrada. Aqui o semáforo é inicializado com valor 1 — uma única "vaga" — e se comporta exatamente como o mutex do cenário 2. A diferença real do semáforo aparece no padrão produtor-consumidor: o sem_post pode ser chamado por uma thread diferente da que chamou sem_wait — o produtor acorda o consumidor — o que o mutex não permite. 

sem_wait(&g_sem);   // "entra": 1 → 0  (bloqueia se já zero)
    bf = pg_read_bal(c, from);
    bt = pg_read_bal(c, to);
    pg_write_bal(c, from, bf - amt);
    pg_write_bal(c, to,   bt + amt);
sem_post(&g_sem);   // "sai":  0 → 1  (acorda a próxima thread)

04 SELECT FOR UPDATE {{SCN_4_BADGES}}
{{SCN_4_DYNAMIC}}

Mecanismo

É como reservar um livro na prateleira antes de pegá-lo: Thread A coloca uma etiqueta de "reservado" nas linhas do banco (FOR UPDATE). Thread B tenta as mesmas linhas, vê a reserva e aguarda. Quando Thread A conclui (COMMIT), a reserva some e Thread B lê os saldos já atualizados — sem possibilidade de conflito. Diferente do mutex, o lock é por linha: threads operando em contas distintas não se bloqueiam mutuamente. O ORDER BY id garante que todas as threads reservam linhas na mesma sequência, evitando deadlock (situação em que A espera B e B espera A ao mesmo tempo, para sempre).

Thread A: BEGIN Thread A: SELECT ... FOR UPDATE → reserva adquirida Thread B: SELECT ... FOR UPDATE → BLOQUEADA (aguarda A) Thread A: UPDATE → COMMIT → reserva liberada Thread B: DESBLOQUEADA — lê balance = 900 (valor correto) Thread B: UPDATE balance = 700 ✓

05 SERIALIZABLE + retry {{SCN_5_BADGES}}
{{SCN_5_DYNAMIC}}

Mecanismo

É como um árbitro observando dois jogadores editando a mesma planilha ao mesmo tempo: se as alterações criariam um resultado inconsistente, o árbitro cancela a última e pede para refazer desde o estado atual. No banco de dados, o PostgreSQL monitora dependências entre transações; ao detectar conflito, aborta uma delas com o código SQLSTATE 40001 (serialization failure). A aplicação captura o erro, faz ROLLBACK e tenta novamente. O número de retentativas (retries) mede diretamente o nível de contenção — quanto mais conflito, mais retrabalho.

06 UPDATE Atômico {{SCN_6_BADGES}}
{{SCN_6_DYNAMIC}}

Mecanismo

A diferença está em onde o cálculo acontece. Na abordagem comum, a aplicação busca o saldo, faz a conta e devolve o resultado — três passos separados que criam uma janela de tempo para interferência:

── Abordagem comum (com janela de corrida) ────────────── Thread A: lê balance = 1000 ← valor vai para a memória da thread Thread A: calcula 1000 - 100 = 900 ← conta feita fora do banco Thread A: grava balance = 900 ← manda o resultado de volta ↑ Thread B pode ler o mesmo 1000 nessa janela ── UPDATE atômico (sem janela) ────────────────────────── Banco: lê + calcula + grava SET balance = balance - 100 tudo em um único passo indivisível, dentro do banco

No UPDATE atômico você não manda um valor fixo de volta — você manda uma instrução: SET balance = balance − 100. O banco lê, calcula e grava tudo internamente, sem que o valor saia para a aplicação em nenhum momento. É como a diferença entre pegar o cofrinho, contar o dinheiro, adicionar as moedas na mão e enfiar tudo de volta versus passar as moedas direto pela fenda: no segundo caso, o cofrinho nunca saiu do lugar e não há janela para interferência. A palavra atômico vem exatamente dessa ideia: a operação é indivisível, ou acontece inteira ou não acontece — nunca pela metade. As contas são atualizadas em ordem crescente de id (lock ordering) para eliminar a possibilidade de deadlock.

Leitura + escrita separadas 4 Sim
SELECT FOR UPDATE 3 Não
UPDATE atômico 2 Não

Por que mutex e semáforo não escalam? O lock neste projeto é global — uma única trava para todas as transferências. Thread A (contas 1→2) bloqueia Thread B (contas 8→9), mesmo sem conflito real. Mais threads significam mais contenção, não mais paralelismo. O SELECT FOR UPDATE e o UPDATE atômico utilizam locks por linha, permitindo que operações sobre contas distintas ocorram em paralelo.

6. Por que o UPDATE Atômico é a Melhor Solução?

Com os dados dos experimentos em mãos, é possível explicar por quê cada mecanismo se comporta como se comporta — e não apenas o que cada um faz.

6.1 O Problema é a Janela de Corrida

Toda transferência bancária tem três etapas: ler o saldo, calcular o novo valor e gravar o resultado. Quando essas etapas são separadas, existe uma janela de tempo entre a leitura e a gravação — e é exatamente nessa janela que as condições de corrida ocorrem. Cada mecanismo tenta fechar essa janela de uma forma diferente.

Mecanismo Queries ao BD Tipo de proteção Escala
Sem controle 4 Nenhuma —
Mutex / Semáforo 4 Lock global (aplicação) Não
SELECT FOR UPDATE 3 Lock por linha (BD) Sim
SERIALIZABLE 3 + retries Detecção de conflito (BD) Parcial
UPDATE atômico 2 Atomicidade (BD) Sim

6.2 Por que Mutex e Semáforo Perdem

Mutex e semáforo usam um lock global: uma única trava para todas as transferências, independentemente de quais contas estão envolvidas. É como um supermercado com dez caixas que exige uma fila única antes da entrada — não importa que os caixas sejam independentes, todos os clientes passam pelo mesmo gargalo. Com mais threads, a contenção aumenta e o desempenho piora em vez de melhorar.

6.3 Por que SELECT FOR UPDATE é Bom, mas Não o Melhor

O SELECT FOR UPDATE traz um avanço importante: o lock é por linha, não global. Threads operando em contas distintas não se bloqueiam — o supermercado agora tem filas separadas por setor. Porém, ainda exige três idas ao banco (SELECT com lock + dois UPDATEs), e o lock fica ativo desde o SELECT até o COMMIT. Durante esse intervalo, qualquer outra thread que precise das mesmas contas fica esperando.

6.4 Por que o UPDATE Atômico Vence

O UPDATE atômico elimina a etapa problemática na raiz: a leitura na aplicação deixa de existir. Em vez de "busque o valor, calcule, devolva", a instrução SET balance = balance − X faz tudo dentro do banco em um único passo indivisível — como passar moedas direto pela fenda do cofrinho em vez de abri-lo, contar e recolocar. Não há valor saindo para a memória da thread, não há janela de tempo entre leitura e gravação, não há nada para outra thread corromper. Resultado: 2 queries por transferência (as menores entre todos os mecanismos), lock de linha que dura microssegundos e escala linear com o número de threads.

Quando usar cada solução UPDATE atômico — operações aritméticas simples: somar, subtrair, incrementar saldo. Sem necessidade de leitura prévia.

SELECT FOR UPDATE — lógica que precisa ler antes de decidir: verificar saldo mínimo, checar limite diário, aplicar regras de negócio antes de gravar.

Mutex / Semáforo — ambientes sem banco de dados relacional, ou quando o número de threads e recursos é pequeno e a simplicidade importa mais que o desempenho.

7. Conclusão

Condições de corrida constituem um dos problemas mais insidiosos em sistemas concorrentes: o programa executa sem erros visíveis, mas os dados produzidos são incorretos. Em sistemas financeiros, isso pode se traduzir em perdas monetárias reais, inconsistências contábeis ou vulnerabilidades exploráveis por usuários maliciosos.

Os experimentos demonstraram que diferentes mecanismos de proteção impõem trade-offs distintos entre corretude, throughput e complexidade de implementação. A escolha adequada depende do contexto: operações aritméticas simples beneficiam-se do UPDATE atômico; lógica de negócio complexa que exige leitura prévia é melhor servida pelo SELECT FOR UPDATE; ambientes sem banco de dados relacional recorrem a mutex ou semáforo.
