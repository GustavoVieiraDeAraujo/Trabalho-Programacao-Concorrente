Race Conditions
in Bank Transfers

Computational Experiment Report

Comparative evaluation of six synchronization mechanisms — POSIX Pthreads primitives at the application layer and transactional control in PostgreSQL — applied to concurrent bank transfers. The experiment quantifies the impact of race conditions and measures the throughput of each solution.

1. Introduction

Imagine two bank tellers simultaneously checking the balance of the same account: both see R$ 1,000, each calculates independently, and both write the result. The second teller overwrites the first — as if the previous operation never existed. This is the central problem of this work: when multiple threads (execution units running in parallel) access the same data without coordination, the outcome depends on the random order in which the operating system schedules them. This phenomenon is called a race condition.

In financial systems, the effect is silent and dangerous: money can be created from nothing or disappear without any error message. The program runs normally, logs show everything is fine — but the balances are wrong. This experiment reproduces that scenario at scale and compares six protection mechanisms.

Table 1 shows what happens when two threads operate on the same account simultaneously, starting from a balance of R$ 1,000.00. Each row represents a moment in time; notice how Thread B reads an already-stale balance.

Table 1 — Example of a lost update: Thread A withdraws R$ 300 and Thread B withdraws R$ 400 concurrently.

Instant Thread A  (−R$ 300) Thread B  (−R$ 400) Actual balance
t₁ Reads balance → R$ 1,000 — R$ 1,000

The correct result would be R$ 300.00 (1,000 − 300 − 400). But Thread B read the balance before Thread A finished — so it wrote as if A's withdrawal never happened. This pattern is called a lost update: R$ 300.00 was created from nothing. The mechanisms in section 3 aim to solve exactly this.

3. Tested Scenarios

Six approaches are evaluated, from the worst case — complete absence of synchronization — to the most efficient — an atomic operation delegated to the database. Scenarios 1 to 3 control concurrency at the application layer (Pthreads); scenarios 4 to 6 delegate control to PostgreSQL.

01 No Control {{SCN_1_BADGES}}
{{SCN_1_DYNAMIC}}

Mechanism

No protection is used — it is like two editors opening the same document simultaneously without version control: both start from the same content, make their changes independently, and whoever saves last overwrites the other's work. The sched_yield() call voluntarily yields the CPU between the read and the write, widening the window in which the two threads can interleave and making data corruption consistently observable in tests.

Thread A: SELECT balance = 1000 (account 3) Thread B: SELECT balance = 1000 (account 3) ← same value! ↑ race window Thread A: UPDATE balance = 900 (subtracted 100) Thread B: UPDATE balance = 800 (should be 700 — overwrote A!)

02 pthread_mutex_t {{SCN_2_BADGES}}
{{SCN_2_DYNAMIC}}

Mechanism

Works like the key to a shared restroom: whoever grabs the key (pthread_mutex_lock) runs the operation; the others wait outside. When finished, it returns the key (pthread_mutex_unlock) and the next thread can enter. Because there is a single key for all accounts — regardless of which accounts a transfer involves — no two operations run in parallel. This guarantees correctness but eliminates parallelism. 

pthread_mutex_lock(&g_lock);    // grab key — blocks if another thread holds it
    bf = pg_read_balance(c, from);
    bt = pg_read_balance(c, to);
    pg_write_balance(c, from, bf - amt);
    pg_write_balance(c, to,   bt + amt);
pthread_mutex_unlock(&g_lock);  // return key — releases for the next thread

Assessment

CriterionAssessmentNote
Correctness ✅ Guaranteed Full mutual exclusion
Scalability Does not scale Global lock serializes independent operations
Production use 🟡 Low contention Suitable for few threads or few accounts
03 sem_t — POSIX Semaphore {{SCN_3_BADGES}}
{{SCN_3_DYNAMIC}}

Mechanism

Think of a parking lot with a vacancy display: a car entering decrements the counter (sem_wait); one leaving increments it (sem_post). If the display reads zero, the car waits at the entrance. Here the semaphore is initialized to 1 — a single "slot" — and behaves exactly like the mutex in scenario 2. The real difference of a semaphore appears in the producer-consumer pattern: sem_post can be called by a different thread from the one that called sem_wait — the producer wakes the consumer — which a mutex does not allow. 

sem_wait(&g_sem);   // "enter": 1 → 0  (blocks if already zero)
    bf = pg_read_balance(c, from);
    bt = pg_read_balance(c, to);
    pg_write_balance(c, from, bf - amt);
    pg_write_balance(c, to,   bt + amt);
sem_post(&g_sem);   // "leave": 0 → 1  (wakes the next thread)

04 SELECT FOR UPDATE {{SCN_4_BADGES}}
{{SCN_4_DYNAMIC}}

Mechanism

It is like reserving a library book before picking it up: Thread A places a "reserved" tag on the database rows (FOR UPDATE). Thread B tries the same rows, sees the reservation, and waits. When Thread A commits, the reservation is released and Thread B reads the already updated balances — with no possibility of conflict. Unlike the mutex, the lock is per row: threads operating on different accounts do not block each other. The ORDER BY id ensures all threads reserve rows in the same sequence, preventing deadlocks (where A waits for B and B waits for A, forever).

Thread A: BEGIN Thread A: SELECT ... FOR UPDATE → lock acquired Thread B: SELECT ... FOR UPDATE → BLOCKED (waiting for A) Thread A: UPDATE → COMMIT → lock released Thread B: UNBLOCKED — reads balance = 900 (correct value) Thread B: UPDATE balance = 700 ✓

CriterionAssessmentNote
Correctness ✅ Guaranteed Row-level lock in the database
Scalability Scales well Threads on different accounts do not block
Production use ✅ Recommended Ideal for complex business logic
05 SERIALIZABLE + retry {{SCN_5_BADGES}}
{{SCN_5_DYNAMIC}}

Mechanism

It is like a referee watching two players edit the same spreadsheet simultaneously: if the changes would produce an inconsistent result, the referee cancels the last one and asks for a redo from the current state. In the database, PostgreSQL monitors dependencies between transactions; upon detecting a conflict, it aborts one of them with SQLSTATE 40001 (serialization failure). The application catches the error, issues a ROLLBACK, and retries. The retry count directly measures the contention level — the more conflicts, the more rework.

Thread A: BEGIN ISOLATION LEVEL SERIALIZABLE Thread B: BEGIN ISOLATION LEVEL SERIALIZABLE Thread A: SELECT 1000 → UPDATE 900 → COMMIT ✓ Thread B: SELECT 1000 → UPDATE 800 → COMMIT ✗ SQLSTATE 40001 Thread B: ROLLBACK → retry → SELECT 900 → UPDATE 700 → COMMIT ✓

CriterionAssessmentNote
Correctness ✅ Guaranteed DB detects and resolves conflicts automatically
Overhead Variable Proportional to the number of retries
Production use 🟡 Low contention Efficient when conflicts are rare
06 Atomic UPDATE {{SCN_6_BADGES}}
{{SCN_6_DYNAMIC}}

Mechanism

The difference lies in where the calculation happens. In the common approach, the application fetches the balance, computes the new value, and sends it back — three separate steps that create a time window for interference:

── Common approach (with race window) ─────────────────── Thread A: reads balance = 1000 ← value goes into thread memory Thread A: computes 1000 - 100 = 900 ← calculation done outside the DB Thread A: writes balance = 900 ← sends the result back ↑ Thread B can read the same 1000 in this window ── Atomic UPDATE (no window) ──────────────────────────── DB: reads + computes + writes SET balance = balance - 100 all in a single indivisible step, inside the database

With the atomic UPDATE you do not send a fixed value back — you send an instruction: SET balance = balance − 100. The database reads, calculates, and writes entirely internally, without the value ever leaving to the application. It is like the difference between picking up the piggy bank, counting the money, adding coins in your hand, and stuffing it all back versus dropping coins directly through the slot: in the second case the piggy bank never left its place and there is no window for interference. The word atomic comes from exactly this idea: the operation is indivisible — it either happens entirely or not at all, never halfway. Accounts are updated in ascending id order (lock ordering) to eliminate the possibility of deadlocks.

Why don't mutex and semaphore scale? The lock in this project is global — a single lock for all transfers. Thread A (accounts 1→2) blocks Thread B (accounts 8→9), even with no real conflict. More threads means more contention, not more parallelism. SELECT FOR UPDATE and the atomic UPDATE use per-row locks, allowing operations on different accounts to run in parallel.

6. Why Is the Atomic UPDATE the Best Solution?

With the experimental data in hand, it is possible to explain why each mechanism behaves as it does — not just what each one does.

6.1 The Problem Is the Race Window

Every bank transfer has three steps: read the balance, compute the new value, and write the result. When these steps are separated, a time window exists between the read and the write — and that is exactly where race conditions occur. Each mechanism tries to close that window in a different way.

Table 5 — Operational cost comparison of each mechanism.

Mechanism DB queries Protection type Scales
No control 4 None —
Mutex / Semaphore 4 Global lock (application) No
SELECT FOR UPDATE 3 Row-level lock (DB) Yes
SERIALIZABLE 3 + retries Conflict detection (DB) Partial
Atomic UPDATE 2 Atomicity (DB) Yes

6.2 Why Mutex and Semaphore Lose

Mutex and semaphore use a global lock: a single latch for all transfers, regardless of which accounts are involved. It is like a supermarket with ten checkout lanes that forces everyone through a single entrance queue — it doesn't matter that the lanes are independent; all customers pass through the same bottleneck. With more threads, contention grows and performance worsens instead of improving.

6.3 Why SELECT FOR UPDATE Is Good, but Not the Best

SELECT FOR UPDATE brings an important improvement: the lock is per row, not global. Threads operating on different accounts do not block each other — the supermarket now has separate queues per section. However, it still requires three round-trips to the database (SELECT with lock + two UPDATEs), and the lock stays active from the SELECT until the COMMIT. During that interval, any other thread needing the same accounts must wait.

6.4 Why the Atomic UPDATE Wins

The atomic UPDATE eliminates the problematic step at the root: the application-side read ceases to exist. Instead of "fetch the value, calculate, send it back", the instruction SET balance = balance − X does everything inside the database in a single indivisible step — like dropping coins directly through the piggy bank slot instead of opening it, counting, and restuffing. No value leaves into thread memory, no time window exists between read and write, nothing for another thread to corrupt. Result: 2 queries per transfer (the fewest of all mechanisms), a row-level lock lasting microseconds, and linear scaling with the number of threads.

When to use each solution Atomic UPDATE — simple arithmetic operations: add, subtract, increment balance. No prior read needed.

SELECT FOR UPDATE — logic that must read before deciding: check minimum balance, verify daily limit, apply business rules before writing.

Mutex / Semaphore — environments without a relational database, or when the number of threads and resources is small and simplicity matters more than throughput.

7. Conclusion

Race conditions are among the most insidious problems in concurrent systems: the program runs without visible errors, but the data it produces is incorrect. In financial systems, this can translate into real monetary losses, accounting inconsistencies, or vulnerabilities exploitable by malicious users.

The experiments demonstrated that different protection mechanisms impose distinct trade-offs between correctness, throughput, and implementation complexity. The right choice depends on context: simple arithmetic operations benefit from the atomic UPDATE; complex business logic that requires a prior read is better served by SELECT FOR UPDATE; environments without a relational database fall back on mutex or semaphore.
